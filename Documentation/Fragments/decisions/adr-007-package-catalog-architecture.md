# ADR-007: Package Catalog Architecture

## Status
Accepted

## Context
iFoundation generates projects with many optional packages (AppStrings, A11yIdentifiers, AppNetworking, BenchmarkKit, DeveloperMenu, etc.). Each package needs to be:
- Toggleable during project creation
- Independently versioned
- Testable in isolation
- Composable with other packages

## Decision
Implement a **Package Catalog** system where each optional package is:

1. A separate Swift Package Manager module
2. Defined by a `PackageManifest` (YAML/JSON)
3. Toggleable via the wizard or CLI flags
4. Auto-wired into the generated project's `Package.swift`

## Package Manifest Schema

```yaml
# Packages/AppStrings/manifest.yml
id: app-strings
name: AppStrings
description: Type-safe localization strings with compile-time safety
category: infrastructure
default: true

dependencies: []

conflicts: []

requires: []

files:
  - source: Package.swift.template
    destination: Packages/AppStrings/Package.swift
  - source: Core/
    destination: Packages/AppStrings/Sources/AppStrings/Core/
  - source: Modules/
    destination: Packages/AppStrings/Sources/AppStrings/Modules/

tests:
  - source: Tests/
    destination: Packages/AppStrings/Tests/

hooks:
  post-install:
    - generate-string-catalogs
  pre-commit:
    - check-localization-coverage

lint-rules:
  - no_hardcoded_strings
  - localization_completeness
```

## Package Categories

| Category | Packages | Purpose |
|----------|----------|---------|
| `infrastructure` | AppStrings, A11yIdentifiers, AppRoutes, AnalyticsEvents | Type-safe cross-cutting concerns |
| `data` | SwiftData, CoreData, CloudKit, Keychain | Persistence and storage |
| `networking` | AppNetworking | HTTP client and API definitions |
| `development` | DeveloperMenu, BenchmarkKit, Logging | Debug and performance tools |
| `testing` | SnapshotTests, PerformanceTests | Extended testing capabilities |
| `ci` | Codecov, DocCPublish | CI/CD integrations |

## Generated Project Structure with Packages

```
MyApp/
├── Package.swift                    # Main package, references all sub-packages
├── Sources/
│   └── MyApp/                       # App target
├── Packages/                        # Local Swift packages
│   ├── AppStrings/                  # Type-safe localization
│   ├── A11yIdentifiers/             # Type-safe accessibility
│   ├── AppNetworking/               # Type-safe networking (if enabled)
│   ├── DeveloperMenu/               # Debug menu (if enabled)
│   └── BenchmarkKit/                # Benchmarking (if enabled)
├── Tests/
│   ├── MyAppTests/
│   └── MyAppUITests/               # Uses A11yIdentifiers
└── .foundation/
    └── config.yml                   # Records which packages were enabled
```

## Generated Package.swift

```swift
// swift-tools-version: 6.3
import PackageDescription

let package = Package(
    name: "MyApp",
    platforms: [.iOS(.v17), .macOS(.v14)],
    dependencies: [
        // Local packages (generated by iFoundation)
        // AppStrings (default: enabled)
        // A11yIdentifiers (default: enabled)
        // AppNetworking (user enabled)
        // DeveloperMenu (user enabled)
    ],
    targets: [
        .executableTarget(
            name: "MyApp",
            dependencies: [
                "AppStrings",
                "A11yIdentifiers",
                "AppNetworking",      // conditional
                "DeveloperMenu",      // conditional
            ]
        ),
        .testTarget(
            name: "MyAppTests",
            dependencies: [
                "MyApp",
                "AppStrings",
                "AppNetworking",      // for mock testing
            ]
        ),
    ]
)
```

## Adding New Packages (For Contributors)

1. Create `Packages/<PackageName>/manifest.yml`
2. Add template files in `Templates/shared/packages/<PackageName>/`
3. Register in `PackageCatalog.swift`
4. Add tests in `Tests/PackageTests/`
5. Update documentation

## Consequences

### Positive
- **Modularity**: Each package is independent
- **Testability**: Packages tested in isolation
- **Discoverability**: Catalog makes features discoverable
- **Extensibility**: Easy to add new packages
- **Versioning**: Each package can evolve independently

### Negative
- **Complexity**: More files and dependencies
- **Build time**: More packages = longer initial build
- **Coordination**: Package updates may need coordination
